# NiumaScene 核心场景制作指南

## 目标
核心场景用于承载全局服务，解决跨场景切换时状态丢失的问题。  
它应该在游戏启动后常驻，不随 RPG 场景、建筑场景、MiniGame 场景卸载。

一句话原则：

```text
核心场景放服务，业务场景放内容。
```

## 为什么需要核心场景
如果 `NiumaSceneController` 跟随普通业务场景一起卸载，会出现：

- `ReturnContext` 丢失，MiniGame 退出时报 `ReturnContextMissing`。
- `NiumaMiniGameController` 被销毁，房间连接、玩家 ID、RoomId 丢失。
- `NiumaAudioController` 被销毁，BGM / 环境音切换中断。
- `NiumaSaveController` 和各模块 SaveAdapter 被销毁，读写档流程不稳定。
- 全局 UI、对话框、Loading UI 在切场景时重复创建或失效。

所以第一版推荐创建一个单独的核心场景，例如：

```text
CoreScene
```

并把它放在 Build Settings 的第一个场景。

## 推荐层级
推荐一个模块或功能集绑定一个物体。不要把所有脚本都堆在同一个 GameObject 上，否则 Inspector 很难维护。

```text
CoreScene
└── BootstrapRoot
    ├── SceneRoot
    │   ├── LoadingRoot
    │   └── SceneAudioRoot
    ├── AudioRoot
    ├── UIRoot
    │   ├── EventSystem
    │   ├── Canvas_Global
    │   └── UIAudioRoot
    ├── SaveRoot
    │   └── SaveAdapters
    ├── GameplayServicesRoot
    └── MiniGameSessionRoot
```

## Inspector 配置阅读规则
后续所有模块 README 都建议按这个规则补充。策划或关卡同学看文档时，应优先确认三件事：

1. 这个字段填什么。
2. 这个字段能不能留空。
3. 留空之后系统会怎么处理。

配置表中的“可留空”含义：

- `可以`：留空不会阻断主流程，脚本会自动查找或跳过对应功能。
- `不建议`：留空通常还能跑，但正式场景容易找错对象或缺少表现。
- `不可以`：正式流程必须填写，否则功能无法执行。

## 核心脚本 Inspector 配置表
### NiumaGameBootstrapper
挂载位置：

```text
BootstrapRoot
```

用途：让核心场景常驻，并在启动后进入第一个正式业务场景。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Bootstrap Root` | 拖 `BootstrapRoot` 自己 | 可以 | 为空时脚本使用当前 GameObject | 指定 `DontDestroyOnLoad` 的常驻根物体 |
| `Dont Destroy On Load` | 正式场景开启 | 不建议关闭 | 关闭后切业务场景会卸载核心服务，返回栈会丢失 | 保证核心服务跨场景常驻 |
| `Destroy Duplicate Bootstrap` | 正式场景开启 | 不建议关闭 | 关闭后重复进入 CoreScene 可能出现两套核心服务 | 防止重复全局控制器 |
| `First Scene Name` | 第一个正式业务场景名，例如 `RPG_Village` | 可以 | 为空时不会自动加载业务场景，只停留在 CoreScene | 控制游戏启动后进入哪个场景 |
| `Load First Scene On Start` | 需要自动进游戏时开启 | 可以 | 关闭后需要外部按钮或脚本手动加载首场景 | 适合主菜单或调试启动流程 |
| `Clear Return Stack Before First Scene` | 正式启动流程开启 | 可以 | 关闭后旧返回栈可能残留，调试时容易误返回旧场景 | 新游戏/主菜单进入时清理返回历史 |
| `Show Loading UI On First Scene` | 有 Loading UI 时开启 | 可以 | 关闭后首场景加载时不显示 Loading 面板 | 控制首场景加载表现 |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 为空时会在 BootstrapRoot 子物体中自动查找，层级复杂时可能找不到 | 显式绑定场景服务入口 |
| `Log Warnings` | 正式调试期建议开启 | 可以 | 关闭后配置错误不会输出警告 | 方便排查启动失败 |

### NiumaSceneController
挂载位置：

```text
BootstrapRoot/SceneRoot
```

用途：全局唯一场景切换服务，负责加载、返回栈、出生点恢复、输入冻结意图和检查点保存意图。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Fallback Scene Name` | 安全兜底场景名，例如 `RPG_Village` 或 `MainMenu` | 可以 | 加载失败时只记录错误，不会自动去兜底场景 | 加载失败时给玩家一个可回退场景 |
| `Max Return Context Depth` | 建议 8 | 不可以 | 太小会导致嵌套返回不足，太大会让返回链难维护 | 限制返回栈深度 |
| `Default Return Overflow Policy` | 建议 `RejectNew` | 不可以 | 策略不合适时可能丢弃旧返回链 | 返回栈满时如何处理新返回点 |
| `Freeze Input During Load By Default` | 正式跨场景加载建议开启 | 可以 | 关闭后切场景期间玩家可能继续移动或交互 | 默认冻结玩家输入 |
| `Show Loading UI By Default` | 有 Loading UI 时开启 | 可以 | 关闭后默认不显示 Loading UI，单次请求仍可覆盖 | 默认加载表现 |
| `Checkpoint Requester Provider` | 拖核心场景 `SceneRoot/CheckpointRequester` 上的 `NiumaSceneSaveCheckpointRequester`，不是拖业务场景对象 | 可以 | 留空时 `RequestCheckpointSave` 只会产生意图，不会真正保存 | 接入 NiumaSave 检查点 |
| `Auto Find Checkpoint Requester` | 正式场景建议关闭并手动绑定 | 可以 | 开启时会自动查找，场景复杂时可能找错 | 方便测试场景快速跑通 |
| `Log Warnings` | 建议开启 | 可以 | 关闭后场景名缺失、返回栈缺失等问题不提示 | 调试配置错误 |

### SceneLoadingStateBridge
挂载位置：

```text
BootstrapRoot/SceneRoot/LoadingRoot
```

用途：读取 `NiumaSceneController.LoadingSnapshot`，驱动 Loading UI，并把加载期间的输入冻结转发给玩家/交互适配脚本。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 为空时可自动查找；正式多场景下不建议依赖自动查找 | 指定加载状态来源 |
| `Auto Find Scene Controller` | 测试场景可开，正式场景建议关 | 可以 | 关闭且未绑定 Controller 时，Loading 和输入冻结不会生效 | 控制是否自动找场景服务 |
| `Loading Receiver Provider` | 拖 `SceneLoadingPanelBridge` 或自制 LoadingPanel 脚本 | 可以 | 留空时只处理输入冻结，不显示 Loading UI | 把加载数据推给 UI |
| `Input Block Target Providers` | 常驻玩家/交互在核心场景时，拖 `TPCSceneInputBlockTarget` / `InteractSceneInputBlockTarget`；玩家/交互在业务场景时可留空并开启自动查找 | 可以 | 留空且关闭自动查找时，加载期间不会冻结对应输入 | 指定需要被冻结的输入系统 |
| `Input Block Reason` | 默认 `SceneLoading` 即可 | 不建议为空 | 为空会让解除阻塞难以追踪来源 | 让输入系统只解除本桥接加的阻塞 |
| `Unblock When Loading Ends` | 建议开启 | 可以 | 关闭后加载结束不会自动解除本桥接加的输入冻结 | 保证切场景后恢复控制 |
| `Log Warnings` | 建议开启 | 可以 | 关闭后绑定错脚本不提示 | 排查桥接绑定错误 |

### 输入冻结目标到底放在哪里
`Input Block Target Providers` 绑定的是“可被 SceneLoading 冻结的输入适配器”，不是场景本身。它有两种正确摆法。

方案 A：玩家和交互系统在核心场景常驻

```text
BootstrapRoot
├── PlayerRoot
│   └── SceneBridge
│       ├── TPCSceneSpawnTarget
│       └── TPCSceneInputBlockTarget
├── InteractionRoot
│   └── SceneBridge
│       └── InteractSceneInputBlockTarget
└── SceneRoot
    └── LoadingRoot
        └── SceneLoadingStateBridge
```

配置方式：

- `TPCSceneInputBlockTarget.Player Controller` 拖 `PlayerRoot` 上的 `PlayerModuleController`。
- `InteractSceneInputBlockTarget.Interaction Controller` 拖 `InteractionRoot` 上的 `NiumaInteractionController`。
- `SceneLoadingStateBridge.Input Block Target Providers` 数组拖这两个适配器。
- `Auto Find Input Block Targets` 可以关闭。

方案 B：玩家和交互系统在业务场景中

```text
RPG_Village
├── PlayerRoot
│   └── SceneBridge
│       └── TPCSceneInputBlockTarget
└── InteractionRoot
    └── SceneBridge
        └── InteractSceneInputBlockTarget
```

配置方式：

- 核心场景 `SceneLoadingStateBridge.Input Block Target Providers` 可以留空。
- 开启 `Auto Find Input Block Targets`。
- 每个需要被冻结输入的业务场景，都在玩家或交互根物体下挂对应适配器。
- 不要在核心场景 Inspector 中跨场景拖业务场景对象，这种引用在 Unity 中不适合作为正式配置。

如果玩家或交互系统是运行时生成的，且生成时间晚于 `sceneLoaded`，生成完成后调用 `SceneLoadingStateBridge.RebuildTargets()`，让 Loading 桥重新扫描输入适配器。

| `Use Unscaled Time` | 建议开启 | 可以 | 关闭后 TimeScale=0 时 Loading 动画可能不动 | 避免暂停状态影响 UI |
| `Disable Root When Hidden` | 建议开启 | 可以 | 关闭后透明面板仍可能参与布局或射线 | 隐藏后禁用面板根节点 |
| `Block Raycasts When Visible` | 建议开启 | 可以 | 关闭后 Loading 期间玩家可能点到后面的 UI | Loading 遮罩是否拦截点击 |
| `Keep Visible On Failure` | 建议开启 | 可以 | 关闭后失败提示会立即消失 | 给玩家/测试人员看到错误 |
| `Failure Visible Seconds` | 建议 3 秒 | 可以为 0 | 为 0 时失败提示不保留 | 控制失败提示停留时间 |

### 检查点保存到底保存哪个场景
检查点保存不是“绑定某个场景然后保存这个场景”。正确理解是：

```text
入口按钮/传送点/对话跳转勾选 RequestCheckpointSave
    -> NiumaSceneController 发现需要保存检查点
    -> 调用核心场景 CheckpointRequester
    -> CheckpointRequester 调用核心场景 NiumaSaveController.SaveCheckpointAsync
    -> NiumaSaveController 收集当前已注册的各模块 SaveAdapter 快照并写入 checkpoint_01
```

因此：

- `NiumaSceneController.Checkpoint Requester Provider`：拖 `SceneRoot/CheckpointRequester` 上的 `NiumaSceneSaveCheckpointRequester`。
- `NiumaSceneSaveCheckpointRequester.Save Controller`：拖 `SaveRoot` 上的 `NiumaSaveController`。
- RPG、MiniGame、建筑内部等业务场景对象不要拖到这里。
- 任务、剧情、玩家位置、背包等奖励是否能保存，取决于对应模块的 SaveAdapter 是否已注册到 `NiumaSaveController.ProviderRegistry`。
### NiumaSceneSaveCheckpointRequester
挂载位置：

```text
BootstrapRoot/SceneRoot/CheckpointRequester
```

用途：当场景切换请求勾选 `RequestCheckpointSave` 时，转发给核心场景里的 `NiumaSaveController` 保存检查点。它只负责转发保存请求，不绑定其他业务场景。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Save Controller` | 拖核心场景 `SaveRoot` 上的 `NiumaSaveController`，不要拖业务场景对象 | 不建议 | 为空时可自动查找；找不到则检查点保存失败，但场景切换仍可继续 | 指定真正执行保存的控制器 |
| `Auto Find Save Controller` | 测试场景可开，正式场景建议关 | 可以 | 关闭且未绑定时无法保存检查点 | 控制是否自动找存档服务 |
| `Checkpoint Display Name Prefix` | 例如 `Scene Checkpoint` / `进入小游戏前` | 可以 | 为空时使用默认 `Scene Checkpoint` | 存档列表里显示的检查点名称前缀 |
| `Write Mode` | 第一版建议 `LocalOnly` | 不可以 | 配置不当可能导致切场景被云同步阻塞 | 控制检查点写本地还是云端策略 |
| `Log Warnings` | 建议开启 | 可以 | 关闭后保存失败不提示 | 排查保存桥接问题 |

### SceneAudioBridge
挂载位置：

```text
BootstrapRoot/SceneRoot/SceneAudioRoot
```

用途：根据加载状态和当前激活场景播放 BGM、环境音和加载音效。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 为空时可自动查找；找不到则不响应加载状态音效 | 读取场景加载状态 |
| `Audio Controller` | 拖 `AudioRoot` 上的 `NiumaAudioController` | 不建议 | 为空时可自动查找；找不到则不播放音频 | 播放 CueId |
| `Load Started Cue` | 填加载开始 CueId | 可以 | 为空时加载开始不播放音效 | 加载开始音效 |
| `Load Completed Cue` | 填加载完成 CueId | 可以 | 为空时加载完成不播放音效 | 加载完成音效 |
| `Load Failed Cue` | 填加载失败 CueId | 可以 | 为空时加载失败不播放音效 | 加载失败音效 |
| `Scene Audio Cues` | 每项填 SceneName、BGM Cue、Ambient Cue | 可以 | 留空时不按场景切 BGM/环境音 | 场景到音频的映射表 |
| `Ambient Channel Id` | 建议 `scene_ambient` | 不建议为空 | 为空时环境音无法播放，但 BGM 不受影响 | 同频道环境音互相替换 |
| `Stop Ambient When Scene Has No Config` | 按需求开启 | 可以 | 关闭后进入未配置场景时保留旧环境音 | 控制无配置场景是否停环境音 |
| `Restart Bgm If Same Scene` | 通常关闭 | 可以 | 开启后重复进入同场景会让 BGM 从头播放 | 控制同场景重复激活时音乐是否重播 |
| `Log Warnings` | 建议开启 | 可以 | 关闭后 CueId 缺失或播放失败不提示 | 排查音频配置 |

### SceneSpawnPoint
挂载位置：

```text
业务场景/SpawnPoint_xxx
```

用途：标记玩家进入场景、返回场景、复活或同场景传送时的落点。

### SceneButtonAction
挂载位置：

```text
业务场景/UI按钮 或 交互入口物体
```

用途：给 Unity Button / UnityEvent 使用，把无返回值按钮事件转发给 `NiumaSceneController`。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 | 配置意义 |
| --- | --- | --- | --- | --- |
| `Scene Controller` | 拖核心场景 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 为空时可自动查找；找不到按钮无效 | 指定场景切换服务 |
| `Target Scene Name` | 填 Build Settings 中场景名 | 加载场景时不可以 | 为空时 `LoadConfiguredScene()` 不执行 | 要进入哪个场景 |
| `Target Spawn Point Id` | 填目标场景出生点 ID | 可以 | 为空时不指定出生点，可能走默认出生点 | 进入目标场景后的落点 |
| `Restore Player At Target Spawn Point` | 需要进场定位时开启 | 可以 | 关闭后 `Target Spawn Point Id` 被忽略 | 是否把玩家放到目标点 |
| `Purpose` | 按用途选择，例如 `MiniGame` / `EnterBuilding` / `Teleport` | 可以 | `None` 会按普通切换处理 | 标记本次切换用途 |
| `Push Return Context` | 需要返回时开启 | 可以 | 关闭后 `ReturnToPreviousScene()` 找不到本次来源 | 是否记录当前场景为返回点 |
| `Return Scene Name` | 通常留空 | 可以 | 留空时自动记录当前激活场景 | 特殊入口强制返回指定场景 |
| `Return Spawn Point Id` | 填返回点 ID，例如 `npc_minigame_return` | PushReturnContext 开启时建议填 | 为空时返回后可能走默认出生点 | 返回时玩家落点 |
| `Clear Return Stack Before Push` | 主菜单进游戏可开 | 可以 | 普通入口误开会清掉旧返回链 | 压入返回点前是否清空旧历史 |
| `Freeze Input During Load` | 正式切场景建议开启 | 可以 | 关闭后加载中玩家可能继续操作 | 加载期间冻结输入 |
| `Show Loading UI` | 跨场景加载建议开启 | 可以 | 关闭后不显示 Loading 面板 | 加载表现 |
| `Request Checkpoint Save` | 进入小游戏/副本/剧情节点可开 | 可以 | 关闭后不请求检查点保存 | 切场景前保存检查点 |
| `Replace Pending Request` | 建议开启 | 可以 | 关闭后连续点击可能保留旧 Pending 请求 | 连续请求时保留最后意图 |

## 最小可运行版本
如果只是先解决场景返回栈丢失，最小版本只需要：

```text
BootstrapRoot
└── SceneRoot
```

### BootstrapRoot
挂载脚本：

- `NiumaGameBootstrapper`

Inspector 建议：

- `Bootstrap Root`：拖 `BootstrapRoot` 自己。
- `Dont Destroy On Load`：开启。
- `Destroy Duplicate Bootstrap`：开启。
- `First Scene Name`：填写第一个正式业务场景，例如 `RPG_Village`。
- `Load First Scene On Start`：开启。
- `Clear Return Stack Before First Scene`：开启。
- `Show Loading UI On First Scene`：有 Loading UI 时开启，没有时可关闭。
- `Scene Controller`：拖 `SceneRoot` 上的 `NiumaSceneController`。

用途说明：

- `BootstrapRoot` 是常驻根物体。
- 业务场景切换时，它不会被卸载。
- 如果重复进入 CoreScene，新的 `BootstrapRoot` 会被销毁，避免出现两套全局服务。

### SceneRoot
挂载脚本：

- `NiumaSceneController`

Inspector 建议：

- `Fallback Scene Name`：填写安全兜底场景，例如 `RPG_Village` 或 `MainMenu`。
- `Max Return Context Depth`：建议 8。
- `Default Return Overflow Policy`：建议 `RejectNew`。
- `Freeze Input During Load By Default`：开启。
- `Show Loading UI By Default`：有 Loading UI 时开启。
- `Checkpoint Requester Provider`：如果接入存档检查点，拖 `NiumaSceneSaveCheckpointRequester`；没有接入可留空。

注意：

- 全项目只保留一份 `NiumaSceneController`。
- RPG 场景、建筑场景、MiniGame 场景不要再挂 `NiumaSceneController`。

## 完整推荐版本
当项目进入正式流程，建议在核心场景中补齐以下节点。

## SceneRoot 相关
### LoadingRoot
挂载脚本：

- `SceneLoadingStateBridge`
- `SceneLoadingPanelBridge`

用途：

- `SceneLoadingStateBridge` 负责接收场景加载状态。
- `SceneLoadingPanelBridge` 负责把加载状态推给 Loading UI。

绑定建议：

- Loading 面板属于全局 UI 时，放在 `UIRoot/Canvas_Global` 下。

### SceneAudioRoot
挂载脚本：

- `SceneAudioBridge`

用途：

- 场景切换时播放 BGM。
- 场景切换时播放环境音。
- 场景加载开始 / 完成时播放音效。

绑定建议：

- `Audio Controller` 绑定 `AudioRoot` 上的 `NiumaAudioController`。
- 场景和 CueId 的映射由策划配置。

### CheckpointRequester
如果需要切场景时请求检查点保存，建议在 `SceneRoot` 下创建：

```text
SceneRoot
└── CheckpointRequester
```

挂载脚本：

- `NiumaSceneSaveCheckpointRequester`

绑定建议：

- `Save Controller` 绑定 `SaveRoot` 上的 `NiumaSaveController`。
- 在 `NiumaSceneController.Checkpoint Requester Provider` 中拖入该组件。这里拖的是核心场景里的桥接器，不是其他业务场景。

## AudioRoot
挂载脚本：

- `NiumaAudioController`

用途：

- 全局 BGM。
- UI 音效。
- 角色 3D 音效。
- 对话音效。
- 场景环境音。

绑定建议：

- `AudioCueDefinition` 配置所有 CueId。
- `AudioCatalogDefinition` 配置 CueId / AddressKey 到 AudioClip 的本地映射。
- 第一版不要求 Addressables，后期可替换 Resolver。

注意：

- 不要在每个业务场景重复放 `NiumaAudioController`。
- 业务模块只通过桥接播放声音。

## UIRoot
建议放：

- `EventSystem`
- `Canvas_Global`
- `UIManager`
- `UIAudioBridge`
- `NiumaUIDialogueViewBridge`

### EventSystem
用途：

- Unity UI Button、InputField、拖拽事件都需要它。

注意：

- 全局 UI 由核心场景管理时，业务场景不要再放第二个 EventSystem。
- 如果某个独立测试场景单独运行，可以临时放一个 EventSystem，正式场景合并时删除。

### UIManager
挂载脚本：

- `UIManager`

用途：

- 注册 / 打开 / 关闭全局 UI View。
- 对话框、Loading、提示面板等可走全局 UI。

绑定建议：

- `UIViewRegistrySO` 中注册常用 ViewId。
- 对话窗口 ViewId 建议保持稳定，例如 `DialogueWindow`。

### UIAudioBridge
挂载脚本：

- `UIAudioBridge`

用途：

- UI 模式切换、面板打开关闭、按钮焦点变化等音效。

绑定建议：

- `Audio Controller` 绑定 `AudioRoot` 上的 `NiumaAudioController`。

### NiumaUIDialogueViewBridge
挂载脚本：

- `NiumaUIDialogueViewBridge`

用途：

- 把 `NiumaGal` 对话 Presenter 接到 `NiumaUI` 的 DialogueWindowView。

绑定建议：

- 如果对话框是全局 UI，放核心场景。
- 如果某个独立剧情场景有专用对话框，也可以放业务场景，但不要两边同时驱动同一个 ViewId。

## SaveRoot
挂载脚本：

- `NiumaSaveController`

建议子节点：

```text
SaveRoot
└── SaveAdapters
```

`SaveAdapters` 下按模块挂：

- `NiumaGalSaveAdapter`
- `NiumaQuestSaveAdapter`
- `NiumaInventorySaveAdapter`
- `NiumaEquipmentSaveAdapter`
- `NiumaCraftSaveAdapter`
- `NiumaRewardSaveAdapter`
- `NiumaAttributeSaveAdapter`
- `NiumaEffectSaveAdapter`
- `NiumaGrowthSaveAdapter`
- `NiumaSkillSaveAdapter`

绑定建议：

- 每个 Adapter 绑定对应模块 Controller。
- 每个 Adapter 绑定 `SaveRoot` 上的 `NiumaSaveController`。
- 如果某个模块还没启用，对应 Adapter 可以先不挂。

注意：

- SaveAdapter 是全局存档桥接，不建议放在业务场景。
- 业务场景卸载不应该导致存档 Provider 丢失。

## GameplayServicesRoot
这里放“游戏事实服务”。这些服务通常需要跨场景保留。

建议挂载：

- `NiumaInventoryController`
- `NiumaEquipmentController`
- `NiumaCraftController`
- `NiumaRewardController`
- `NiumaQuestController`
- `NiumaAttributeController`
- `NiumaEffectController`
- `NiumaGrowthController`
- `NiumaSkillController`
- `NiumaDialogueController`

是否必须全部放核心场景：

- 已经做成全局事实的模块，放核心场景。
- 只在某个测试场景临时使用的模块，可以先放测试场景。
- 后期进入正式整合后，再迁移到核心场景。

绑定建议：

- 背包、装备、合成、奖励之间互相有依赖，尽量在 Inspector 中手动绑定 Controller。
- 任务奖励由 `NiumaRewardQuestBridge` 负责时，`NiumaRewardController` 和 `NiumaQuestController` 都建议放核心场景。
- 属性、效果、技能、成长互相关联时，也建议都放核心场景，避免场景切换后状态断开。

## MiniGameSessionRoot
如果 MiniGame 只有一个场景，并且退出时直接断开连接，可以暂时不放核心场景。  
如果 MiniGame 有多个子场景，例如：

```text
MiniGame_StartScene -> MiniGame_GameplayScene -> MiniGame_StartScene
```

建议在核心场景放：

- `NiumaMiniGameController`
- `MiniGameUIViewBridge`

用途：

- 保持 TCP / UDP 连接。
- 保持 PlayerId、RoomId、Host、Viewer 状态。
- 防止切换 MiniGame 子场景时被服务器判定断线。

注意：

- MiniGame UI 可以放在各自业务场景。
- MiniGameController 如果放核心场景，业务场景里的 UI 只绑定或自动查找它。

## 业务场景还需要放什么
业务场景不再放全局 Controller，但仍要放场景内容。

### RPG 场景
建议放：

- 玩家预制体。
- NPC。
- 交互物。
- 任务触发物。
- 场景内摄像机、灯光、建筑、地形。
- `SceneSpawnPoint`：每个可返回点 / 传送点一个。
- NPC 对话入口脚本。
- 局部 UI 入口按钮。

```text
└── SceneBridge
```

`SceneBridge` 挂：

- `TPCSceneSpawnTarget`

挂：

- `InteractSceneInputBlockTarget`

用途：

- 场景加载期间冻结玩家输入。
- 场景加载期间冻结交互输入。
- 场景加载完成后恢复玩家到指定 SpawnPoint。

### MiniGame 开始场景
建议放：

- MiniGame 开始界面 Canvas。
- `MiniGameStartScreenUI`。
- `MiniGameUIViewBridge`：如果不放核心场景，可以放在开始场景。
- 房间大厅 UI、取名 UI、房间号输入 UI。

注意：

- 如果 `NiumaMiniGameController` 放核心场景，开始场景里不要再放第二个。

### MiniGame 游戏中场景
建议放：

- 游戏中 UI Canvas。
- 画板 UI。
- 聊天 UI。
- 答案输入 UI。
- 玩家列表 UI。
- 回合倒计时 UI。
- 评价 UI。
- `MiniGameGameplaySceneReturnBridge`：如果玩法结束后需要自动回到房间 / 开始场景。

注意：

- 游戏结束统一回房间，不直接回 RPG。
- 退出游戏按钮才回 RPG。

### 建筑 / 室内场景
建议放：

- 室内内容。
- 入口 / 出口 `SceneSpawnPoint`。
- 出口按钮或门交互脚本。

## 常见错误
### ReturnContextMissing
原因：

- 没有从带 `PushReturnContext` 的入口进入。
- `NiumaSceneController` 跟随业务场景被卸载。
- 没有核心场景常驻。

解决：

- 确保 `BootstrapRoot` 开启 `DontDestroyOnLoad`。
- 确保 `NiumaSceneController` 放核心场景。
- 进入 MiniGame / 建筑时开启 `PushReturnContext`。

### 按钮无反应
检查：

- 场景是否有 `EventSystem`。
- Canvas 是否有 `GraphicRaycaster`。
- 鼠标是否被 TPC 锁定。
- Button OnClick 是否绑定了正确方法。
- 对应脚本是否绑定 Controller。

### UI 或音频重复
原因：

- 多个业务场景都放了 `UIManager` / `NiumaAudioController` / `EventSystem`。

解决：

- 全局 UI 和音频放核心场景。
- 业务场景只放局部 UI。

### 存档 Provider 丢失
原因：

- SaveAdapter 放在业务场景，切场景后被销毁。

解决：

- `NiumaSaveController` 和 SaveAdapter 放核心场景。

## 推荐检查清单
进入正式测试前逐项检查：

- `CoreScene` 在 Build Settings 第 0 个。
- `BootstrapRoot` 挂 `NiumaGameBootstrapper`。
- `Dont Destroy On Load` 已开启。
- `SceneRoot` 挂 `NiumaSceneController`。
- RPG / MiniGame / 建筑场景没有重复的 `NiumaSceneController`。
- 全局音频只存在一个 `NiumaAudioController`。
- 全局 UI 只存在一个 `UIManager` 和一个 `EventSystem`。
- 所有返回点都有稳定 `SceneSpawnPoint.SpawnPointId`。
- MiniGame 入口开启 `PushReturnContext`。
- MiniGame 退出按钮调用返回上一场景，不是直接 `SceneManager.LoadScene`。
- 所有正式按钮都绑定桥接脚本，不直接调用底层返回值方法。

## 团队约定
- 核心场景由程序维护，策划只改必要配置。
- 业务场景由关卡 / UI / 策划摆放内容。
- 新模块如果需要跨场景保留，先放到 `GameplayServicesRoot`。
- 新 UI 如果是全局面板，放 `UIRoot`；如果只属于某个玩法场景，放业务场景。
- 禁止业务场景直接使用 `SceneManager.LoadScene`，统一走 `NiumaSceneController` 或桥接脚本。
