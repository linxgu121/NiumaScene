# NiumaScene 核心场景制作指南

## 目标
核心场景用于承载跨场景常驻服务，解决切换 RPG、建筑、MiniGame、战斗测试场景时的状态丢失问题。

一句话原则：

```text
核心场景放全局服务，业务场景放内容、玩家、NPC、交互物和局部表现。
```

从 NiumaUI 2.1.0 起，正式 UI 主线是 **UI Toolkit**。核心场景文档不再推荐旧 UGUI `UIManager / DefaultViewFactory / DialogueWindowView / DialogueWindowBinding`。旧场景如果还有这些 Missing Script，应删除后改接 `UIToolkitUIManager + UIToolkitViewFactory + ToolkitReceiver + ToolkitBindingProvider`。

## 制作顺序
建议按下面顺序搭建，不要一口气把所有脚本都挂完：

1. 创建 `CoreScene`，并把它放到 Build Settings 的第 0 个。
2. 创建 `BootstrapRoot`，挂 `NiumaGameBootstrapper`，开启 `Dont Destroy On Load`。
3. 创建 `SceneRoot`，挂场景切换、Loading、检查点保存相关脚本。
4. 创建 `AudioRoot`，挂全局音频服务。
5. 创建 `UIRoot`，先完成 `UIToolkitUIManager / UIToolkitViewFactory / UIDocument`。
6. 创建 `SaveRoot`，挂 `NiumaSaveController` 和各模块 SaveAdapter。
7. 创建 `GameplayServicesRoot`，挂背包、任务、属性、成长、效果、技能、商店、奖励等服务 Controller。
8. 如果小游戏需要跨场景不断线，创建 `MiniGameSessionRoot` 并放 `NiumaMiniGameController`。
9. 最后再打开各业务场景，放玩家、NPC、交互、出生点、门、按钮等局部物体。

## 推荐完整层级
推荐一个模块或功能集绑定一个物体。不要把所有脚本都堆在同一个 GameObject 上。

```text
CoreScene
└── BootstrapRoot                         [NiumaGameBootstrapper]
    ├── SceneRoot                         [NiumaSceneController]
    │   ├── LoadingRoot                   [SceneLoadingStateBridge]
    │   ├── SceneAudioRoot                [SceneAudioBridge]
    │   └── CheckpointRequester           [NiumaSceneSaveCheckpointRequester]
    ├── AudioRoot                         [NiumaAudioController]
    ├── UIRoot
    │   ├── EventSystem                   [EventSystem + UI 输入模块]
    │   ├── UIToolkitRoot
    │   │   ├── UIDocument_HUD            [UIDocument]
    │   │   ├── UIDocument_Prompt         [UIDocument]
    │   │   ├── UIDocument_Dialogue       [UIDocument]
    │   │   ├── UIDocument_Menu           [UIDocument]
    │   │   ├── UIDocument_Popup          [UIDocument]
    │   │   ├── UIDocument_Loading        [UIDocument]
    │   │   └── UIDocument_Debug          [UIDocument]
    │   ├── UIManager                     [UIToolkitUIManager + UIToolkitViewFactory]
    │   ├── UIBridges                     [各模块 UIViewBridge / ToolkitReceiver]
    │   └── BindingProviders              [各模块 ToolkitBindingProvider]
    ├── SaveRoot                          [NiumaSaveController]
    │   └── SaveAdapters                  [各模块 SaveAdapter]
    ├── GameplayServicesRoot              [各模块 Controller]
    └── MiniGameSessionRoot               [NiumaMiniGameController]
```

说明：`UIRoot/UIManager` 是物体名，不是旧 UGUI `UIManager` 脚本。这个物体上挂 `UIToolkitUIManager` 和 `UIToolkitViewFactory`。

## Inspector 配置阅读规则
配置表中的“可留空”含义：

| 标记 | 含义 |
| --- | --- |
| 可以 | 留空不会阻断主流程，脚本会自动查找或跳过对应表现。 |
| 不建议 | 留空通常还能跑，但正式场景容易找错对象或缺少表现。 |
| 不可以 | 正式流程必须填写，否则功能无法执行。 |

## 一、BootstrapRoot
挂载脚本：`NiumaGameBootstrapper`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Bootstrap Root` | 拖 `BootstrapRoot` 自己 | 可以 | 为空时使用当前 GameObject |
| `Dont Destroy On Load` | 正式核心场景开启 | 不建议关闭 | 关闭后切场景会卸载核心服务，返回栈、音频、存档、UI 都会丢 |
| `Destroy Duplicate Bootstrap` | 正式场景开启 | 不建议关闭 | 重复进入 CoreScene 时可能出现两套全局服务 |
| `First Scene Name` | 第一个业务场景名，如 `RPG_Village` | 可以 | 为空时启动后停留在 CoreScene |
| `Load First Scene On Start` | 需要自动进游戏时开启 | 可以 | 关闭后要通过按钮或脚本手动加载首场景 |
| `Clear Return Stack Before First Scene` | 正式启动流程开启 | 可以 | 关闭后上一次编辑器测试留下的返回栈可能残留 |
| `Show Loading UI On First Scene` | 已配置 Loading View 时开启 | 可以 | 关闭后首场景加载不显示 Loading |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 自动查找失败时无法进入首场景 |

## 二、SceneRoot
### SceneRoot 本体
挂载脚本：`NiumaSceneController`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Fallback Scene Name` | 安全兜底场景名，如 `RPG_Village` / `MainMenu` | 可以 | 加载失败时只记录错误，不自动回兜底场景 |
| `Max Return Context Depth` | 建议 8 | 不可以 | 太小会导致嵌套返回不足 |
| `Default Return Overflow Policy` | 建议 `RejectNew` | 不可以 | 策略不当可能丢返回链 |
| `Freeze Input During Load By Default` | 正式跨场景加载建议开启 | 可以 | 加载期间玩家可能继续移动或交互 |
| `Show Loading UI By Default` | 有 Loading View 时开启 | 可以 | 默认不显示 Loading |
| `Checkpoint Requester Provider` | 拖 `SceneRoot/CheckpointRequester` 上的 `NiumaSceneSaveCheckpointRequester` | 可以 | 勾选检查点保存时只产生意图，不真正写档 |
| `Auto Find Checkpoint Requester` | 测试可开，正式建议手动绑定 | 可以 | 关闭且未绑定时检查点保存不可用 |

全项目只保留一个 `NiumaSceneController`。RPG、建筑、MiniGame 业务场景不要重复挂。

### LoadingRoot
挂载脚本：`SceneLoadingStateBridge`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 自动查找失败时无法读取加载状态 |
| `Loading Receiver Provider` | 正式拖 `UIRoot/UIBridges/SceneLoadingToolkitBridge` | 可以 | 留空时只冻结/解冻输入，不显示 Loading UI |
| `Input Block Target Providers` | 如果玩家常驻，拖 `PlayerRoot/SceneBridge/TPCSceneInputBlockTarget` 与 `InteractSceneInputBlockTarget`；如果玩家在业务场景，保持自动查找 | 可以 | 留空且关闭自动查找时不冻结对应输入 |
| `Auto Find Input Block Targets` | 玩家和交互在业务场景时开启 | 可以 | 关闭且未手动绑定时不冻结业务输入 |
| `Input Block Reason` | 默认 `SceneLoading` | 不建议为空 | 为空时难以按来源释放阻塞 |
| `Unblock When Loading Ends` | 建议开启 | 可以 | 关闭后加载结束不会自动释放本桥接加的输入阻塞 |

`TPCSceneInputBlockTarget` 和 `InteractSceneInputBlockTarget` 不是核心场景必备物体。它们通常挂在玩家预制体的 `PlayerRoot/SceneBridge` 上，让 LoadingRoot 自动查找；只有玩家也做成 DontDestroyOnLoad 常驻时，才在核心场景里手动拖引用。

### SceneAudioRoot
挂载脚本：`SceneAudioBridge`

用途：根据场景或场景标签切换 BGM、环境音、加载音效。它不自己播放 AudioClip，只调用 `NiumaAudioController` 注册的音频服务。

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Scene Controller` | 拖 `SceneRoot` 上的 `NiumaSceneController` | 不建议 | 无法跟随场景切换自动播放 |
| `Audio Controller` | 拖 `AudioRoot` 上的 `NiumaAudioController` | 不建议 | 自动查找失败时无声音 |
| `Scene Cue Sets` | 每个场景配置 BGM / Ambient / Loading CueId | 可以 | 未配置的场景不自动换音乐 |
| `Ambient ChannelId` | 建议 `scene_ambient` | 不建议为空 | 为空时只跳过环境音，不应阻断 BGM |

### CheckpointRequester
挂载脚本：`NiumaSceneSaveCheckpointRequester`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Save Controller` | 拖 `SaveRoot` 上的 `NiumaSaveController` | 不建议 | 检查点保存不会真正执行 |
| `Auto Find Save Controller` | 测试可开，正式建议手动绑定 | 可以 | 关闭且未绑定时检查点保存失败 |

正确链路：

```text
NiumaSceneController.Checkpoint Requester Provider
    -> SceneRoot/CheckpointRequester/NiumaSceneSaveCheckpointRequester
        -> SaveRoot/NiumaSaveController
            -> SaveRoot/SaveAdapters/各模块 SaveAdapter
```

业务场景只负责在入口按钮、传送点或对话跳转中勾选 `Request Checkpoint Save`。

## 三、AudioRoot
挂载脚本：`NiumaAudioController`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Cue Definitions` | 拖音效、BGM、UI、语音、环境音配置 | 不建议 | CueId 无法解析，播放请求失败 |
| `Audio Catalog` | 拖本地 AudioClip 映射表 | 第一版不建议 | AddressKey 找不到 Clip |
| `Register Service To Context` | 核心场景开启 | 可以关闭 | 其它模块无法通过 GameContext 播放音频 |
| `Drive Tick In Update` | 没有统一模块 Tick 时开启 | 可以 | 如果外部统一 Tick，关闭以避免重复驱动淡入淡出 |

建议把所有 CueId 命名写进音频配置表，例如：`ui_click`、`scene_loading_start`、`bgm_village_day`、`gal_choice_show`。业务脚本只填 CueId，不直接拖 AudioClip。

## 四、UIRoot
### EventSystem
挂载脚本：Unity `EventSystem` + 项目使用的 UI 输入模块。

| 情况 | 怎么处理 |
| --- | --- |
| 使用 Unity 新输入系统 | 配 `InputSystemUIInputModule` |
| 使用旧输入系统 | 配 `StandaloneInputModule` |
| 场景里已有 EventSystem | 核心场景保留一个，其它业务场景删除重复 EventSystem |

### UIToolkitRoot / UIDocument 分层
每个 `UIDocument_xxx` 挂 Unity `UIDocument`。这些物体负责承载不同 UI 层，不写业务逻辑。

| 物体 | 建议 Panel / UXML 用途 | 常见 View |
| --- | --- | --- |
| `UIDocument_HUD` | 常驻 HUD 层 | 资源条、任务追踪、TPC 准心、Buff 图标 |
| `UIDocument_Prompt` | 轻提示层 | 交互提示、环境叙事气泡 |
| `UIDocument_Dialogue` | 对话层 | Gal 对话窗口、选项 |
| `UIDocument_Menu` | 菜单层 | 背包、商店、合成、存档、奖励、小游戏大厅 |
| `UIDocument_Popup` | 弹窗层 | Confirm、Toast、错误提示 |
| `UIDocument_Loading` | Loading 遮罩层 | 场景加载进度 |
| `UIDocument_Debug` | 调试层 | 开发面板、诊断窗口 |

### UIManager
挂载脚本：`UIToolkitUIManager`、`UIToolkitViewFactory`

`UIToolkitUIManager` 字段：

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `View Factory` | 拖同物体上的 `UIToolkitViewFactory` | 不建议 | 无法打开任何 Toolkit View |
| `Input Blocker Provider` | 有常驻玩家时拖 `PlayerRoot/UIBridge/TPCGameplayInputBlocker`；没有常驻玩家可先留空 | 可以 | 菜单/对话/Loading 打开时不会冻结玩家控制 |
| `Toast / Confirm / Loading ViewId` | 填注册表中的通用 ViewId | 可以 | 对应通用弹窗不可用 |
| `Enable Keyboard Back` | 需要 ESC 返回时开启 | 可以 | 关闭后需要外部输入系统调用返回 |
| `Drive Tick In Update` | 没有统一模块 Tick 时开启 | 可以 | 外部统一 Tick 时应关闭，避免重复驱动 |

`Input Blocker Provider` 拖的是 `TPCGameplayInputBlocker`，不是 `SceneLoadingStateBridge` 的 Input Block Target。前者是 UI 打开菜单/对话时冻结玩家控制，后者是场景加载期间冻结玩家/交互。

`UIToolkitViewFactory` 字段：

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Registry` | 拖 `UIToolkitViewRegistrySO` | 不建议 | 无法按 ViewId 创建 View |
| `Layer Roots` | 每层拖对应 `UIDocument` 的 Root | 不建议 | 对应层无法显示 View |
| `Binding Provider Behaviours` | 拖 `BindingProviders` 下的各模块 Provider | 可以 | 未匹配时使用空 Binding，窗口可能打开但无内容 |
| `Log Warnings` | 建议开启 | 可以 | ViewId、Layer、Provider 配错时不提示 |

### UIToolkitViewRegistrySO 常用注册表
创建方式：Project 右键 `Create / NiumaUI / Toolkit View Registry`。

| ViewId | LayerId | BindingProviderId | InputPolicy | 用途 |
| --- | --- | --- | --- | --- |

## 五、SaveRoot
### SaveRoot 本体

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |

### 服务初始化建议顺序
如果脚本由 Unity Awake/Start 自动初始化，仍建议在 Hierarchy 中按依赖顺序摆放，方便检查：

```text
Attribute -> Inventory -> Quest -> Gal -> Story -> Growth -> Effect -> Skill -> Equipment -> Craft -> Shop -> Reward
```

实际服务多数支持自动解析，但正式场景推荐手动拖引用，减少“找到了错误场景对象”的风险。

## 七、MiniGameSessionRoot
挂载脚本：`NiumaMiniGameController`

| 字段 | 怎么填 | 可留空 | 不填/关闭会怎样 |
| --- | --- | --- | --- |
| `Register Service To Context` | 需要其它模块打开小游戏时开启 | 可以关闭 | Gal / Scene / UI 无法通过 GameContext 找到小游戏服务 |
| `Network Client / Mock Client` | 联机阶段绑定真实客户端，单机调试绑定 Mock | 不可以 | 无法创建房间、加入房间、同步状态 |
| `Auto Connect` | 需要进入场景自动连接时开启 | 可以 | 关闭后需要开始 UI 主动连接 |

如果小游戏跨多个场景，`NiumaMiniGameController` 建议放核心场景常驻，避免切场景断开 TCP/UDP。小游戏具体 UI 可以放 `UIRoot`，或在 MiniGame 场景加载后由 Toolkit View 显示。

## 八、业务场景还需要放什么
核心场景不是把所有东西都搬进去。下面这些仍然应该放业务场景或玩家预制体。

| 放置位置 | 需要放什么 | 说明 |
| --- | --- | --- |
| RPG / 建筑 / 室内场景 | `SceneSpawnPoint` | 每个场景至少一个默认出生点；返回 MiniGame/RPG 时靠它定位 |
| 门、传送点、按钮 | `SceneButtonAction` | Inspector 按钮不直接调 `SceneManager.LoadScene`，统一走场景服务 |
| 玩家根物体 `PlayerRoot` | `NiumaCharacterController`、`PlayerModuleController` | TPC 动作、输入、角色模块事实 |
| `PlayerRoot/SceneBridge` | `TPCSceneInputBlockTarget`、`InteractSceneInputBlockTarget` | 让 LoadingRoot 可以冻结玩家控制和交互 |
| `PlayerRoot/UIBridge` | `TPCGameplayInputBlocker`、`TPCPlayerHudToolkitBridge` | UI 打开时冻结玩法输入；推送准心/控制提示 |
| `PlayerRoot/AudioBridge` | `TPCAudioBridge` | 把 TPC 跳跃、落地、受伤音效转给 NiumaAudio |
| `PlayerRoot/InteractionRoot` | `NiumaInteractionController`、Detector、PromptBridge | 玩家交互检测和提示，不建议放核心场景 |
| NPC、拾取物、商店柜台、机关 | 对应 `IInteractable` 脚本 | 内容对象跟随业务场景加载/卸载 |
| MiniGame 入口 NPC | `MiniGameDialogueActionHandler` 或对应 Gal ActionHandler | 对话选项进入小游戏，仍走 SceneController |
| 拾取物 | `InventoryPickupInteractable` | 只属于当前场景内容，不放核心场景 |
| 商店 NPC / 柜台 | `ShopInteractable` 或项目自定义交互入口 | 打开核心场景里的 Shop UI |

## 九、对话选项触发场景跳转
对话选项进入小游戏或建筑时，不建议直接在按钮 UI 事件中写跳转。推荐：

1. `DialogueAsset` 的选项填写稳定 `ChoiceId`，如 `enter_draw_guess`。
2. `NiumaDialogueController.Action Handler Provider` 绑定 `MiniGameDialogueActionHandler` 或项目自定义 `IDialogueActionHandler`。
3. Handler 内部调用 `NiumaSceneController.LoadScene`，并设置 `PushReturnContext = true`。
4. 退出小游戏时调用 `ReturnToPreviousScene()`，返回对话前的 RPG 场景落点。

这样不会绕过 Gal 的选择记录、任务推进和存档事实。

## 十、常见错误
### ReturnContextMissing
入口没有开启 `PushReturnContext`，或 `NiumaSceneController` 跟随业务场景被卸载。解决：确认 `NiumaSceneController` 在 CoreScene 常驻，入口脚本开启返回上下文。

### Loading 不显示
- `SceneLoadingStateBridge.Loading Receiver Provider` 没拖 `SceneLoadingToolkitBridge`。
- `SceneLoadingToolkitBridge.UI Manager` 没绑定 `UIToolkitUIManager`。
- `UIToolkitViewRegistrySO` 没注册 `Loading` View。
- `UIToolkitViewFactory.Layer Roots` 没配置 Loading 层的 `UIDocument`。
- `UIToolkitViewFactory.Binding Provider Behaviours` 没拖 `ToolkitLoadingBindingProvider`。

### UI 窗口打开但没有内容
- 对应 `XxxToolkitBindingProvider` 没拖进 `UIToolkitViewFactory.Binding Provider Behaviours`。
- `UIToolkitViewRegistrySO` 的 `BindingProviderId` 和 Provider 脚本里的 ID 不一致。
- UXML 元素 name 和 BindingProvider 字段不一致。

### 对话时鼠标不能点选项
- `NiumaGalToolkitDialogueViewBridge` 的 Cursor Mode 没设置为选项出现时显示。
- TPC 锁鼠标逻辑没有被对话桥接接管。
- 准心/HUD 没加入 `Hide Objects During Dialogue`。

### UI 或音频重复
多个业务场景重复放 `UIToolkitUIManager`、`NiumaAudioController` 或 `EventSystem`。正式流程应把全局 UI、音频、存档放核心场景。

### 检查点保存没有生效
- `NiumaSceneController.Checkpoint Requester Provider` 没拖 `NiumaSceneSaveCheckpointRequester`。
- `NiumaSceneSaveCheckpointRequester.Save Controller` 没拖 `NiumaSaveController`。
- 对应模块 SaveAdapter 没注册到 `NiumaSaveController`。

## 十一、最终检查清单
- `CoreScene` 在 Build Settings 第 0 个。
- `BootstrapRoot` 挂 `NiumaGameBootstrapper`，并开启 `Dont Destroy On Load`。
- `SceneRoot` 挂 `NiumaSceneController`，并绑定 `CheckpointRequester`。
- `LoadingRoot` 挂 `SceneLoadingStateBridge`，并绑定 `SceneLoadingToolkitBridge`。
- `AudioRoot` 只有一个 `NiumaAudioController`。
- `UIRoot` 只有一个 EventSystem。
- `UIRoot/UIManager` 只挂 `UIToolkitUIManager` 与 `UIToolkitViewFactory`。
- `UIToolkitViewRegistrySO` 注册 Loading、DialogueWindow、InteractionPrompt、InventoryPanel 等正式 View。
- `UIToolkitViewFactory.Layer Roots` 指向各 `UIDocument` 的 Root。
- `UIToolkitViewFactory.Binding Provider Behaviours` 已拖入所有启用模块的 Provider。
- `SaveRoot` 挂 `NiumaSaveController`，SaveAdapters 都绑定了对应 Controller。
- `GameplayServicesRoot` 的 Controller 开启 `Register Service To Context`。
- 玩家预制体有 `TPCGameplayInputBlocker`、`TPCSceneInputBlockTarget`、`InteractSceneInputBlockTarget`。
- 业务场景入口统一走 `SceneButtonAction` 或业务桥接，不直接调用 `SceneManager.LoadScene`。
